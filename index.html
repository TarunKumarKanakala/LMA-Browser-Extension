01 What this actually is

This zip is only the browser-extension half of AWS's sample solution "Live Meeting Assistant" (LMA). It is a Chrome (Manifest V3) extension that sits in a meeting tab, watches the meeting UI to figure out who's talking, captures raw audio, and streams both up to an AWS backend over a WebSocket. It contains no speech recognition, no voice-print recognition, and no AI itself — all of that lives in a separate backend CloudFormation stack that this extension only connects to via a URL baked into lma_config.json at build time.

Two completely separate mechanisms are easy to conflate:
1) Audio capture — uses the browser's screen/tab-share permission dialog, purely to grab tab audio.
2) Speaker-name detection — reads the meeting app's own on-screen "who's talking" highlight from the DOM.
They run independently and are not derived from one another.

03 Chrome extension internals

Manifest V3 extension, side panel UI. Three separate JavaScript execution contexts matter:

ContextFilesCan seeJob
Content script teams.js, zoom.js, webex.js, meet.js, chime.js The meeting page's DOM (not its JS variables) Scrape names, watch for speaking highlight, mute state, participant join/leave
Injected page script zoom-injection.js etc. The page's actual JS runtime (e.g. a global MeetingConfig var) Pull data content scripts can't reach, relay via window.postMessage
Hidden recorder iframe recorder.html, recorder.js, audio-worklet.js Nothing about the meeting UI — pure media APIs Capture mic + tab audio, encode to PCM, hand off to the side panel
Side panel (React) src/*.tsx Nothing about the page directly — only what's relayed via chrome.runtime Auth, UI, own the WebSocket, format + send everything to AWS

service-worker.js is 4 lines — its only job is telling Chrome to open the side panel when the toolbar icon is clicked. It does no message routing itself.

Mic + tab audio → PCM → WebSocket

    SidePanel->>CS: chrome.runtime "StartTranscription"
    CS->>IFrame: relay "StartTranscription"
    IFrame->>Browser: getDisplayMedia({video:true, audio:true})
    Note right of Browser: Native "Choose what to share"
picker — ONLY used to grab
tab/system audio, video ignored
    Browser-->>IFrame: displayStream (tab audio)
    IFrame->>Browser: getUserMedia({audio:true})
    Browser-->>IFrame: micStream (your mic)
    IFrame->>IFrame: convert both to mono,
merge into 2-channel signal
(ch0 = mic, ch1 = tab audio)
    IFrame->>IFrame: AudioWorklet: float32 → PCM16,
interleave channels
    IFrame->>SidePanel: chrome.runtime "AudioData"
(base64 PCM)
    SidePanel->>SidePanel: decode base64,
apply mute/pause (zero channel)
    SidePanel->>WS: raw binary PCM frame
Sample rate requested: 8000 Hz. Channel 0 = your microphone. Channel 1 = tab/system audio (everyone else's voice, as played through your speakers). This is why the "share screen" prompt appears even though nobody is trying to see your screen.

Teams Live (teams-live.js, for teams.live.com) uses the identical technique with different CSS class names (fui-Primitive ___19upu4n / ___s78zj80) because the consumer Teams UI ships different internals, and reads your own name from localStorage["msal.activeUserProfile"] instead of clicking an avatar button.

How "you" (host) differ from the other 5 people

WhoHow identifiedSent as
You (the extension user) Scraped once via your own avatar/profile element (Teams: me-control-avatar-trigger → fallback chain of 3 more selectors), or typed manually into the form agentId — fixed for the whole call
Other participants Whichever name text is attached to the tile currently showing the "speaking" highlight, re-read fresh every time the highlight moves activeSpeaker — overwritten on every SPEAKER_CHANGE event

There is no persistent participant ID or roster tracking who the "5 others" are as a set — only a rolling "whoever is highlighted right now" string, sent as a timestamped event each time it changes.

07 Wire format — what actually goes over the WebSocket

// Text frame — sent on start, on every speaker change, and on stop
{
  "callEvent": "SPEAKER_CHANGE",     // START | SPEAKER_CHANGE | END
  "agentId": "<your name>",
  "fromNumber": "+9165551234",       // hardcoded placeholder (call-center data model reused)
  "toNumber": "+8001112222",         // hardcoded placeholder
  "callId": "<Meeting Topic> - <yyyy-mm-dd-hh:mm:ss.mmm>",
  "samplingRate": 8000,
  "activeSpeaker": "Jane Doe"
}

// Binary frames — continuous, interleaved 16-bit PCM
// channel 0 = your mic, channel 1 = tab/system audio, 8kHz

Auth travels as query params on the socket URL itself, not as a header (WebSocket handshakes can't carry custom auth headers from a browser):

wss://<wssEndpoint>?authorization=Bearer <access_token>&id_token=<id_token>&refresh_token=<refresh_token>

09 AWS services — confirmed vs. inferred
